# Data ingestion into Snowflake RAW

## Scope

Loads the five already-processed local datasets into Snowflake RAW landing tables. No dataset is downloaded or generated by this phase; only files already produced per `docs/uci_cleaning_rules.md` and `docs/data_readiness.md` are loaded, as declared by `contracts/*.yml`:

| Contract | Source file | Target table |
|---|---|---|
| `contracts/uci_sales.yml` | `data/processed/uci_sales_clean.parquet` | `RAW.UCI_SALES` |
| `contracts/uci_returns.yml` | `data/processed/uci_returns_clean.parquet` | `RAW.UCI_RETURNS` |
| `contracts/uci_invalid_price.yml` | `data/quarantine/uci_invalid_price.parquet` | `RAW.UCI_SALES_QUARANTINE` |
| `contracts/dim_store.yml` | `data/processed/dim_store_seed.csv` | `RAW.DIM_STORE_SEED` |
| `contracts/dim_product.yml` | `data/processed/dim_product_seed.csv` | `RAW.DIM_PRODUCT_SEED` |

Quarantined rows land in their own table (`RAW.UCI_SALES_QUARANTINE`) and are never blended into `RAW.UCI_SALES` or `RAW.UCI_RETURNS`.

## Design

- **Contract-driven**: `scripts/load_raw_data.py` reads column names and types from `contracts/*.yml` rather than hardcoding them twice; `08_raw_tables.sql`'s columns are cross-checked against the same contracts by `tests/test_snowflake_sql_contracts.py::test_raw_tables_match_contract_columns`, so contract and DDL cannot silently drift apart.
- **Table DDL** (`infra/snowflake/08_raw_tables.sql`) runs under `PHARMARETAIL_ADMIN` as part of the existing BAU deploy pipeline (like `04_grants.sql`/`06_validation.sql`) — no ACCOUNTADMIN bootstrap needed, since `PHARMARETAIL_ADMIN` already owns the `RAW` schema.
- **Data movement** is a separate, manually-dispatched workflow (`snowflake-data-load.yml`), kept apart from the schema-only BAU deploy because moving ~1M rows is a heavier, more deliberate operation than a grants/validation reconcile. It reuses the `SVC_PHARMARETAIL_CICD` service identity from Phase 1 (ADR-002) — no new identity was introduced.
- **RAW enforces structural guarantees, not business rules**: columns the contract marks `nullable: false` are `NOT NULL` in the DDL, so an unexpected null fails the load loudly. Business-rule checks (`accepted_values`, ranges, uniqueness) are deliberately deferred to the dbt/staging layer in a later phase — RAW's job is to be a faithful, audited landing zone, not a validator.
- **Idempotent by truncate-and-reload**: each dataset's target table is `TRUNCATE`d immediately before its `COPY INTO`, in the same run. Re-running the load workflow reproduces the same end state as long as the source files haven't changed. This means a run that fails mid-`COPY` leaves that table empty until the next successful run — there is no automatic rollback to a prior good state, consistent with this project's existing "no automatic rollback" philosophy (`docs/snowflake_runbook.md`). `RAW.LOAD_AUDIT` still records the failed attempt.
- **Audit trail lives in Snowflake**: `RAW.LOAD_AUDIT` gets one row per dataset per run — `load_id`, `source_file`, `file_sha256`, `source_row_count`, `loaded_row_count`, `row_count_match`, `null_counts` (JSON), `duplicate_row_count`, `load_status`, `error_message`, `started_at`, `completed_at` — regardless of whether the run succeeded.
- **Reconciliation report**: after every run, `scripts/load_raw_data.py` writes `docs/data_load_reconciliation.md` (source vs. loaded row counts, duplicates, non-zero null columns, checksum, status per dataset) and the workflow uploads it as a build artifact.

## Pre-load validation, computed locally per dataset

- Row count
- Per-column null counts
- Duplicate-row count (full-row duplicates)
- SHA-256 checksum of the source file

These are compared against the post-`COPY` `SELECT COUNT(*)` from the target table. A mismatch marks the dataset `ROW_COUNT_MISMATCH` in the audit table and fails the workflow (exit code 1) without crashing the run — the other datasets in the batch still get their own audit row via a `finally` block per dataset.

## Running the load

1. Confirm `08_raw_tables.sql` has been applied to the target environment (part of the normal BAU deploy — see `docs/snowflake_runbook.md`).
2. Trigger **Snowflake Data Load** via `workflow_dispatch`, selecting `target_environment`.
3. Review the uploaded `data-load-reconciliation-<environment>` artifact and/or query `RAW.LOAD_AUDIT` directly:

   ```sql
   SELECT table_name, source_row_count, loaded_row_count, row_count_match, load_status, completed_at
   FROM RAW.LOAD_AUDIT
   ORDER BY completed_at DESC;
   ```

Local dry-run (no Snowflake connection required for the pre-load stats):

```bash
python3 -c "
from pathlib import Path
from scripts.load_raw_data import load_contract, compute_file_stats, DATASET_TABLES
for path, table in DATASET_TABLES.items():
    stats = compute_file_stats(load_contract(Path(path)))
    print(table, stats.row_count, stats.duplicate_row_count)
"
```

## Known limitations

- No incremental/CDC loading — this is a full truncate-and-reload per run, appropriate for static demo/seed datasets, not a continuously-arriving source.
- `RAW.UCI_SALES_QUARANTINE`'s `quarantine_reason` currently only ever takes the value `non_positive_price`, per `docs/uci_cleaning_rules.md`.
- `dim_store_seed`'s `postcode` is genuinely null for 30 of 100 rows (see `docs/data_readiness.md`); `contracts/dim_store.yml` was corrected from an earlier `nullable: false` declaration that did not match the actual seed.
